# Next-Level Built-In Prompt Rendering

## Design Goal

Upgrade the built-in model-visible prompt so it is richer, clearer, and more behaviorally effective without changing the underlying cache-friendly architecture.

The prompt must:

- keep the current `Stable Prefix + additive events` model
- preserve `Harness Contract + Identity Envelope + Session Baseline`
- render the stable prefix as high-quality markdown
- improve decision guidance for action selection and user-facing response behavior
- avoid regressing into a monolithic, mutable system prompt

## Current State

The current renderer in `fathom-server/src/agent/prompt/render.rs` is structurally correct, but too sparse:

- `Harness Contract` is a short rule list
- `Identity Envelope` and `Participant Envelope` are rendered as raw JSON dumps
- `Capability Surface` is correct but not very decision-oriented
- `Event Transcript` remains compact and machine-like

The architecture is already strong and cache-friendly. The upgrade target is prompt quality and rendering quality, not a rework of prompt invalidation or stable-prefix design.

## Agreed Architecture

The built-in prompt remains:

1. `Harness Contract`
2. `Identity Envelope`
3. `Session Baseline`
4. `Event Transcript`

The stable prefix remains:

- `Harness Contract`
- `Identity Envelope`
- `Session Baseline`

`Event Transcript` remains additive and outside the stable prefix.

## Rendering Model

### 1. Harness Contract

`Harness Contract` should become a markdown-first operating manual for the runtime semantics layer.

It should explain:

- what the model is allowed to emit
- how to choose between direct response and action execution
- how execution modes work
- how evidence and payload lookups should be used
- what state may or may not be assumed
- how to react to rejection and failure events
- how to avoid unnecessary execution chaining

It should not define persona, tone, backstory, or roleplay identity. That remains the responsibility of `Identity Envelope`.

Recommended internal sectioning:

- `# Harness Contract`
- `## Purpose`
- `## Allowed Outputs`
- `## Choosing The Next Move`
- `## Execution Rules`
- `## Evidence Rules`
- `## State Assumptions`
- `## Failure Handling`
- `## Response Discipline`

### 2. Identity Envelope

`Identity Envelope` should stop rendering as raw JSON in the prompt.

Instead, it should render as:

- metadata header
- markdown material body

Recommended structure:

- `# Identity Envelope`
- `schema_version`
- `source_revision`
- `## Identity Material`
- fenced markdown block containing the identity material

The internal runtime envelope may remain structured. This decision affects prompt rendering, not storage format.

### 3. Session Baseline

`Session Baseline` should remain structured and markdown-rendered.

Its sections remain:

- `## Session Anchor`
- `## Capability Surface`
- `## Participant Envelope`

`Capability Surface` should be rendered for decision-making, not just raw listing. It must preserve:

- environment id and name
- optional environment description
- canonical action ids
- action descriptions
- `mode_support`
- recipe titles and steps

### 4. Participant Envelope

`Participant Envelope` should follow the same prompt-facing rendering direction as `Identity Envelope`.

It should render as:

- metadata header
- markdown material body

Recommended structure:

- `## Participant Envelope`
- `schema_version`
- `source_revision`
- `### Participant Material`
- fenced markdown block containing participant material

### 5. Event Transcript

`Event Transcript` is intentionally not redesigned in this discussion.

The current additive transcript model remains in place.

Its future rendering redesign is deferred to:

- `docs/todos/2026-03-09.13.design-event-transcript-rendering-upgrade.md`

## Draft Harness Contract

The following draft captures the agreed direction for the next-level built-in prompt:

```md
# Harness Contract

## Purpose

You operate inside a session runtime that provides:

- a stable session prefix
- an additive event transcript
- a capability surface of callable actions

Your job is to choose the next best move for the session.

You may emit:

- a direct assistant response to the user
- one or more action executions
- both in the same turn when that is the best next move

## Allowed Outputs

- Use only actions listed in the Session Baseline capability surface.
- Use canonical action ids in the format `env__action`.
- Provide exact action arguments that match the runtime-enforced schema.
- For optional arguments, omit fields you do not need.
- Never send empty placeholder strings for optional fields.

## Choosing The Next Move

- Prefer the smallest sufficient next move.
- If the available evidence is already sufficient, answer the user directly.
- If more information is needed, choose the actions that reduce uncertainty most directly.
- Do not chain executions reflexively when a direct response is already justified.
- Use action execution when the user request requires real inspection, retrieval, or state change.

## Execution Rules

- Execution requests default to `await`.
- Request `detach` only when an action's `mode_support` is `await_or_detach`.
- If an action is `await_only`, requesting `detach` will be rejected.
- Use `detach` only when the current turn does not need that result to decide the next move.
- Multiple executions may be emitted in the same turn.

## Evidence Rules

- Treat execution previews and transcript events as evidence.
- Use resolved payload lookups when present before issuing additional payload fetches.
- Prefer previews first; fetch larger payload slices only when they are necessary for the next decision.
- Avoid redundant payload fetches when equivalent evidence is already present.

## State Assumptions

- Do not assume current time unless an execution result or event provides it explicitly.
- Do not assume live environment state unless an execution result or event provides it explicitly.
- Treat the Session Baseline as the durable contract for this prompt.
- Treat additive events as authoritative updates after the baseline.

## Failure Handling

- `execution_rejected` means the runtime did not accept the requested execution; revise the request instead of assuming it ran.
- `awaited_execution_failed` and `detached_execution_failed` mean execution was accepted but ended unsuccessfully.
- Use the failure message and any payload preview to decide whether to retry, inspect further, change approach, or report failure.

## Response Discipline

- Be direct and useful.
- Do not restate the prompt contract unless it is relevant.
- Do not describe your capabilities unless the user asks.
- Do not over-explain internal execution mechanics unless they matter to the user.
```

## Invariants

- The current stable-prefix architecture must not be weakened.
- No ambient current-time injection should be reintroduced.
- No ambient live-environment-state injection should be reintroduced.
- `Harness Contract` remains runtime semantics only.
- Identity and participant material become markdown-first in prompt rendering.
- Event transcript redesign is deferred, not bundled into this upgrade.

## Rejected Alternatives

### 1. Giant Monolithic Prompt

Rejected because it would:

- blur stable and mutable context
- reduce cache friendliness
- make prompt invalidation harder to reason about

### 2. Keeping Identity and Participants as Raw JSON in Prompt Output

Rejected because:

- the long-term source of truth is expected to be markdown-like authored material
- markdown is more natural for persona and participant context
- raw JSON is less legible for both model and human debugging

### 3. Redesigning Event Transcript In The Same Slice

Rejected for now because:

- the transcript model is already structurally sound
- the stable-prefix prompt upgrade is independently valuable
- transcript rendering deserves a separate design pass focused on token/readability tradeoffs

## Open Issues

- Whether `Capability Surface` should gain slightly richer prose per environment, beyond the current structured listing
- Final wording polish for the `Harness Contract` draft during implementation

4. Expand `Harness Contract` into the agreed multi-section markdown block.
5. Keep `Event Transcript` behavior unchanged in this slice.
