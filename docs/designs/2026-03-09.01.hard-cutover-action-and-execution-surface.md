# Hard Cutover to Action and Execution Surface

## Design Goal

Retire transitional `tool` and `task` compatibility surfaces and hard-cut the canonical model to:

- `Action` for a callable capability
- `Execution` for a runtime instance of an action invocation

This cutover applies to public protocol, runtime, history, prompt, and client vocabulary.

## Agreed Model

### Canonical Nouns

- `Action` is the static capability exposed by an environment.
- `Execution` is the lifecycle record for one requested action run.
- `execution_requested` is the canonical transcript-side invocation record.
- `execution_update` is the canonical runtime-side trigger/update family.

### Non-Canonical Nouns

- `Tool` may remain only inside provider-bridge internals where an upstream API forces that term.
- `Task` is not an acceptable public/runtime/history/client/protocol noun after cutover.

## Hard Cutover Rule

Do not preserve `Task` as a compatibility alias.

Existing `Task*` surfaces are migration debt and should be renamed or removed.

This includes:

- proto messages and enums
- runtime/session APIs
- history event names
- prompt transcript labels
- diagnostics naming
- system environment help text
- client tabs and event rendering

## Target Rename Boundary

The target public/runtime rename sweep is:

- `Task` -> `Execution`
- `TaskStatus` -> `ExecutionStatus`
- `TaskStateChangedEvent` -> `ExecutionStateChangedEvent`
- `TaskDoneTrigger` -> remove
- `ListTasks` -> `ListExecutions`
- `CancelTask` -> `CancelExecution`
- `task_id` -> `execution_id`
- task-payload lookup naming -> execution-payload lookup naming

Prompt and event language should also converge:

- capability surface exposes `actions`, not `tools`
- transcript records use `execution_requested`
- runtime update triggers use `execution_update`

## Cleanup Buckets

The audit identified four buckets:

1. dead-code leftovers
2. legacy trigger/history compatibility
3. task-vs-execution API debt
4. profile-envelope adapter debt

The central cutover bucket is `task-vs-execution API debt`. The other buckets should now be evaluated against the new `Action` / `Execution` vocabulary.

## Key Invariants

- No mixed `Task` / `Execution` public vocabulary after cutover.
- No mixed `Tool` / `Action` canonical vocabulary after cutover.
- `Tool` may exist only as provider-adapter terminology.
- `TaskDoneTrigger` should not survive the cutover.
- Prompt, history, runtime, proto, and client naming should all describe the same execution model.

## Rejected Alternatives

- Keep `Task` as a transport/runtime alias while adopting `Execution` elsewhere.
  - Rejected because mixed nouns prolong compatibility debt and obscure the actual model.
- Keep `Tool` in the canonical prompt/runtime model.
  - Rejected because `Action` is the correct static noun and `Tool` is only an upstream provider term.

## Open Issues

- Exact implementation slicing for the rename sweep.
- Whether low-risk dead-code removals should land before or alongside the `Task` -> `Execution` cutover.
- Whether the profile-envelope cleanup should remain deferred until after the execution-surface cutover.

## Next Implementation Slice

Start with the public/runtime rename boundary:

1. rename proto and runtime APIs from `Task` to `Execution`
2. rename session/client event surfaces from `TaskStateChanged` to `ExecutionStateChanged`
3. remove `TaskDoneTrigger`
4. rename task/history/payload lookup wording to execution wording
5. update client tabs/views from task-focused naming to execution-focused naming

Only after that should remaining dead-code and compatibility leftovers be cleaned up opportunistically.
