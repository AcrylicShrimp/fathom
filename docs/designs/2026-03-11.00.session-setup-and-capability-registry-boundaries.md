# Session Setup And Capability Registry Boundaries

## Design Goal

Separate capability catalog concerns from session setup policy and application bootstrap.

The current runtime mixes:

- capability domain cataloging
- default capability installation
- session setup defaults
- per-session engaged capability selection
- initial capability snapshot construction
- domain-specific setup overrides such as workspace `base_path`

The immediate target is to introduce clean setup layers without changing the external session-creation API yet.

## Current Problems

### Registry owns application composition

`fathom-server/src/capability_domain/registry.rs` currently hard-codes the installed capability domains and builds a global singleton registry.

This means adding or removing a capability domain requires editing registry internals instead of changing a composition root.

### Registry owns execution-routing knowledge

The same module also dispatches execution with a hard-coded match over capability-domain ids. That makes the registry depend on each domain's execution convention.

### Session creation owns setup policy inline

`fathom-server/src/runtime/sessions.rs` currently performs all of the following directly:

- agent profile selection
- participant user selection and dedup
- default engaged capability-domain selection
- initial capability snapshot creation
- domain-specific initial state overrides
- `SessionState` construction

This is already a setup pipeline, but it is unnamed and not swappable.

## Agreed Architecture

Introduce a distinct session setup layer and shrink the capability registry back to catalog responsibilities.

### 1. Capability Catalog

The capability registry should become a catalog/data structure, not a bootstrap root.

Responsibilities:

- store installed capability domains
- store registered actions
- canonicalize and resolve action ids
- expose capability summaries and initial snapshot templates
- support session-scoped action filtering

Non-responsibilities:

- deciding which domains are installed by default
- deciding which domains a session engages
- patching per-session initial state
- provider-specific schema shaping
- execution dispatch branching by hard-coded domain id

Preferred construction shape:

- `CapabilityDomainRegistry::from_capability_domains(...)`

`new()` may remain only as a thin convenience wrapper if it delegates to a separate bootstrap layer.

### 2. Capability Bootstrap

Introduce a bootstrap/composition layer that decides which capability domains are installed in this application build/runtime.

Responsibilities:

- build the default installed capability-domain set
- own any `OnceLock` or process-global default registry wiring

This layer is allowed to be hard-coded for now.

### 3. Session Setup Policy

Introduce an internal setup-policy layer that resolves a session from a thin request into a fully materialized setup.

Proposed stages:

1. `SessionSetupRequest`
2. `SessionSetupPolicy`
3. `SessionSetupResolved`
4. `SessionStateFactory`

#### SessionSetupRequest

Represents external session-creation input.

For now it can stay equivalent to current API input:

- `agent_id`
- `participant_user_ids`

Normalization rule:

- request normalization such as participant-id dedup should happen at the setup boundary, not inside unrelated runtime/session modules

#### SessionSetupPolicy

Resolves setup defaults and selection policy.

Responsibilities:

- resolve agent profile copy
- resolve participant profile copies
- choose engaged capability domains
- derive initial capability snapshots for the session
- apply session-scoped initial state overrides such as workspace `base_path`

For now there will be a single implementation:

- `DefaultSessionSetupPolicy`

This implementation may be hard-coded, but the hard-coding must live here rather than inside registry code or `Runtime::create_session`.

#### SessionSetupContext

The setup policy should depend on a narrow async context trait rather than directly on `Runtime` internals.

Suggested responsibilities for the context trait:

- fetch or create agent profile copy
- fetch or create user profile copy
- provide workspace root
- provide capability catalog access
- allocate next session id

The initial implementation can be a thin adapter over `Runtime`.

This keeps the setup policy testable without requiring a full runtime instance and prevents the setup layer from becoming another dumping ground for unrelated runtime behavior.

#### SessionSetupResolved

Represents the fully materialized setup result.

Expected contents:

- `agent_id`
- `participant_user_ids`
- `agent_profile_copy`
- `participant_user_profiles_copy`
- `engaged_capability_domain_ids`
- `capability_domain_snapshots`

Optional later additions:

- setup diagnostics
- policy revision/version
- install/source metadata
- explicit `created_at_unix_ms` if session creation timestamp should be decided before `SessionState`

#### SessionStateFactory

Constructs `SessionState` from resolved setup data.

This should be a narrow state-construction boundary, not a place where selection policy is reintroduced.

The factory may remain a simple `SessionState::from_setup(...)` if no richer abstraction is needed, but the important rule is that setup decisions must already be finished before that call.

### 4. Execution Controls Layer

Harness-added execution controls such as `execution_mode` should remain outside the capability catalog.

This boundary was already proven by the recent validation bug: harness controls leaking into capability-domain validation caused a boundary failure.

Responsibilities:

- schema augmentation for harness control fields
- control-field validation
- control-field parsing
- control-field stripping/normalization

This layer should not live inside the long-term capability catalog module.

### 5. Execution Dispatcher

Capability execution routing should be separated from the registry.

The near-term goal is:

- move the hard-coded domain-id dispatch match out of the registry

The longer-term goal is:

- register executor handles/adapters alongside capability domains so dispatch is data-driven rather than branch-driven

## Proposed Internal Interfaces

These names are suggestive, not mandatory, but the contracts should be close to this shape.

```rust
pub(crate) struct SessionSetupRequest {
    pub(crate) agent_id: String,
    pub(crate) participant_user_ids: Vec<String>,
}

pub(crate) struct SessionSetupResolved {
    pub(crate) session_id: String,
    pub(crate) agent_id: String,
    pub(crate) participant_user_ids: Vec<String>,
    pub(crate) agent_profile_copy: pb::AgentProfile,
    pub(crate) participant_user_profiles_copy: HashMap<String, pb::UserProfile>,
    pub(crate) engaged_capability_domain_ids: BTreeSet<String>,
    pub(crate) capability_domain_snapshots: HashMap<String, CapabilityDomainSnapshot>,
}

#[async_trait]
pub(crate) trait SessionSetupContext {
    async fn get_or_create_agent_profile(&self, agent_id: &str) -> pb::AgentProfile;
    async fn get_or_create_user_profile(&self, user_id: &str) -> pb::UserProfile;
    fn workspace_root(&self) -> &Path;
    fn capability_registry(&self) -> CapabilityDomainRegistry;
    fn next_session_id(&self) -> String;
}

#[async_trait]
pub(crate) trait SessionSetupPolicy {
    async fn resolve(
        &self,
        ctx: &dyn SessionSetupContext,
        request: SessionSetupRequest,
    ) -> Result<SessionSetupResolved, Status>;
}
```

Important note:

- `next_session_id()` may alternatively remain on `Runtime::create_session` if id allocation should be strictly outside policy; either choice is acceptable as long as ownership is explicit

## Proposed Runtime Flow

Target orchestration shape for `Runtime::create_session`:

1. Build `SessionSetupRequest` from RPC input.
2. Ask the configured setup policy to resolve it.
3. Build `SessionState` from the resolved setup.
4. Spawn the session actor.
5. Store the session runtime handle.
6. Return `SessionSummary`.

In this flow, `Runtime::create_session` becomes orchestration only. It should no longer contain default engagement rules or domain-specific initial-state patching.

## Default Policy Behavior For Now

The default hard-coded setup policy is expected to preserve current behavior:

- dedup participant ids
- create/fetch agent profile copy
- create/fetch participant profile copies
- engage the current default capability-domain set
- build initial capability snapshots from the installed catalog
- patch workspace `base_path` into filesystem and shell capability snapshots

This preserves behavior while moving policy out of the wrong layers.

## Key Invariants

- A capability registry is a catalog, not an application composition root.
- Session setup policy decides engaged capability domains, not the registry.
- Session setup policy may be hard-coded initially, but it must be isolated behind a named layer.
- `SessionState` construction must consume resolved setup data rather than silently performing selection policy.
- Harness execution controls are not capability-domain arguments.
- Capability-domain installation and capability-domain engagement are separate concepts.

## Boundaries

### Registry Boundary

Input:

- installed capability domains

Output:

- action resolution
- summaries
- snapshot templates
- action inventory

Excluded:

- session defaults
- runtime profile fetching
- workspace-specific state patching

### Session Setup Boundary

Input:

- session request
- profile store/runtime access
- capability catalog
- workspace/runtime defaults

Output:

- resolved session setup package

Excluded:

- agent turn execution
- prompt assembly
- action dispatch

### Bootstrap Boundary

Input:

- process/application defaults

Output:

- installed capability-domain set
- default catalog wiring

Excluded:

- per-session decisions

## Tradeoffs

### Accepted

- Keep a hard-coded default implementation for now.

Reason:

- the immediate need is boundary clarity, not dynamic plugin loading

### Accepted

- Keep the external session-creation API unchanged for the first slice.

Reason:

- the internal layering can be improved before exposing richer configuration

### Rejected

- Keep registry self-registering domains because there is no registration pass yet.

Reason:

- lack of a plugin system does not justify mixing catalog and composition responsibilities

### Rejected

- Push all session-setup decisions into `Runtime::create_session`.

Reason:

- that preserves the same coupling under a different filename

## Open Issues

- Whether bootstrap should be owned by `Runtime` construction or a separate capability module.
- Whether execution dispatch should move directly to registered executor adapters in the first refactor slice, or first be extracted into a separate hard-coded dispatcher.
- Whether provider-specific tool schema shaping should move together with execution controls in the same slice or a follow-up slice.
- Whether `created_at_unix_ms` should be assigned inside `SessionState` or inside resolved setup.
- Whether capability snapshot initialization should be exposed as immutable templates plus per-session override hooks.

## Concrete Next Implementation Slice

1. Introduce `SessionSetupRequest`, `SessionSetupResolved`, and `DefaultSessionSetupPolicy`.
2. Move current session-creation default selection logic out of `Runtime::create_session` into that policy.
3. Introduce `CapabilityDomainRegistry::from_capability_domains(...)`.
4. Move default capability installation into a bootstrap function/module outside the registry.
5. Update `Runtime::create_session` to orchestrate:
   - request
   - resolve setup
   - build state
   - spawn actor
6. Leave execution dispatch and execution-control extraction for the next refactor slice unless they fall out mechanically from the first extraction.

## Rejected Alternative

### Put session setup policy directly on `Runtime`

Rejected because:

- it keeps policy and orchestration fused
- it remains awkward to test
- it does not give a clean seam for future non-default setup strategies
