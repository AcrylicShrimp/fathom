# Agent Prompt, Context, and Execution Model

## Design Goal

Rework the model-visible harness contract so the agent reasons over a cache-friendly baseline plus append-only events, instead of a mixed prompt bundle that rebuilds stable and mutable state together every turn. The design must support:

- durable session and capability context
- structured conversation and action/execution history
- minimal operational noise
- principled compaction and baseline rebuilds

## Implementation Note

The implemented harness rebuilds `Harness Contract`, `Identity Envelope`, and `Session Baseline` directly from authoritative `SessionState` and runtime metadata on each turn snapshot. Session compaction summarizes only the append-only event log; summary blocks do not feed baseline reconstruction.

## Current Problems

The current prompt path mixes durable and mutable data in the same assembly path. In particular:

- stable identity and capability material is bundled together with per-turn runtime context
- current time is injected ambiently instead of being explicitly inspected
- resolved lookup payloads and other operational details enter the prompt too eagerly
- runtime execution state is not modeled as a clean append-only event stream for the agent

## Agreed Model

### 1. Stable Prefix Plus Event Log

The model-visible context is split into:

- `Stable Prefix`
- append-only `Delta / Transcript Events`

There is no separate mutable turn blob in the final design.

### 2. Stable Prefix

The stable prefix for the current prompt epoch contains:

- `Harness Contract`
- `Identity Envelope`
- `Session Baseline`

The stable prefix is rebuilt from authoritative runtime state, not from model-produced summaries.

#### Harness Contract

`Harness Contract` is a thin, non-negotiable runtime semantics layer.

It should explain only:

- execution legality
- `await` / `detach` semantics
- batch-settle semantics
- event interpretation rules
- state-assumption rules
- baseline and delta trust rules

It should not define persona, tone, background, motivations, or style. Those belong in `Identity Envelope`.

#### Identity Envelope

`Identity Envelope` is an opaque, versioned, deterministic payload generated by a dedicated agent-profile layer.

The prompt builder consumes the envelope abstraction, not raw source files or copied prompt fragments.

`Identity Envelope` owns all agent-specific behavior definitions, including:

- persona
- role
- tone
- background
- motivations
- worldview
- stylistic behavior
- roleplay-specific identity constraints

#### Session Baseline

`Session Baseline` contains durable session truth:

- `Session Anchor`
- `Capability Surface`
- `Participant Envelope`

#### Session Anchor

This is the minimal explicit session anchor:

- `session_id`
- `started_at`

#### Capability Surface

This is an explicit, structured list describing what the session can do.

Each environment record contains:

- `id`
- `name`
- optional `description`
- `actions`
- `recipes`

Each action entry contains:

- `action_id`
- `description`
- `mode_support`

Each recipe entry contains:

- `title`
- `steps`

`action_id` is the canonical invocation name used by the agent. `mode_support` tells the agent whether `detach` is legal for that action.

The agreed `mode_support` values are:

- `await_only`
- `await_or_detach`

#### Participant Envelope

`Participant Envelope` is an opaque, versioned, deterministic payload generated by a dedicated participant/profile layer.

Like `Identity Envelope`, its internal structure is intentionally deferred and should remain flexible.

### 3. Contract Deltas

When durable contract data changes after the stable prefix was created, do not mutate the old prefix in-place. Instead, append contract delta events.

Examples:

- participant profile updated
- engaged environments changed
- agent profile refreshed

These deltas are append-only. During compaction or stable-prefix re-materialization, superseded deltas are removed and a fresh prefix is rebuilt from authoritative state.

### 4. Conversation / Execution Transcript

The append-only event log also carries the ordinary transcript and execution outcomes.

Core transcript items:

- `user_message`
- `assistant_message`
- `execution_requested` records
- execution outcome records
- contract delta records

### 5. Time and Live CapabilityDomain State

Do not inject current time into the prompt by default.

If the agent needs current time, it should obtain it through an explicit inspection capability rather than an ambient prompt field.

Do not inject live environment state by default either. CapabilityDomain state is considered too volatile to belong in the stable prefix. It should enter context only through:

- explicit inspection actions
- execution results
- trigger items emitted by the runtime

## Action / Execution Model

### 1. Execution Modes

Each `execution_requested` record can request one of two modes:

- `await`
- `detach`

`await` is the default. `detach` must be explicitly specified.

The runtime validates whether the requested mode is allowed by the action contract.

`await` is always the default. `detach` is only legal when the action's `mode_support` is `await_or_detach`.

### 2. Execution Batch

A single agent turn may emit multiple `execution_requested` records. Those requests form one execution batch.

The harness executes the whole batch, then resumes the agent once after the batch settles.

Batch settle rule:

- every `await` execution must be terminal
- every `detach` execution must be either accepted or rejected

Detached jobs do not delay the batch past submission.

### 3. Model-Visible Execution Outcomes

The agent should not see queueing or running-state chatter. Internal states like queued, mapped, dispatched, and running are runtime/UI details, not core reasoning input.

The agreed model-visible execution outcomes are:

- `awaited_execution_succeeded`
- `awaited_execution_failed`
- `execution_detached`
- `detached_execution_succeeded`
- `detached_execution_failed`
- `execution_rejected`

These use the same noun, `execution`, distinguish the two execution modes cleanly, and keep rejection separate from execution failure.

### 4. Event Payload Principles

`awaited_execution_succeeded` includes:

- `execution_id`
- `action_id`
- `payload_preview`

`awaited_execution_failed` includes:

- `execution_id`
- `action_id`
- `message`
- optional `payload_preview`

`execution_detached` includes:

- `execution_id`
- `action_id`

No duplicated args or large details are needed because the original call is already present in history.

`detached_execution_succeeded` includes:

- `execution_id`
- `action_id`
- `payload_preview`

`detached_execution_failed` includes:

- `execution_id`
- `action_id`
- `message`
- optional `payload_preview`

`execution_rejected` includes:

- `execution_id`
- `action_id`
- `message`

`execution_rejected` is intentionally simpler than failure events. The agent already has the original call in transcript, so a concise message is sufficient to explain why the runtime would not accept the call.

### 5. Wakeup Policy

Do not wake the agent on intermediate execution status changes.

Instead:

- resume once after the current execution batch settles
- later detached completions are emitted as future trigger items
- detached completion triggers should be coalesced before waking the agent

This avoids wakeup storms when many executions are in flight concurrently.

## Canonical Terms

- `Action` is the canonical noun for a callable capability exposed by an environment.
- `Execution` is the canonical noun for one runtime instance of an action invocation.
- `Tool` is not part of the canonical harness, prompt, or protocol model. It may remain only inside provider-bridge internals where an upstream API forces that terminology.
- `execution_requested` is the canonical transcript record for an agent-requested action invocation.
- `execution_update` is the canonical trigger/update family for runtime-reported execution outcomes.
- `Task` is not an acceptable public, runtime, history, client, or protocol noun after hard cutover. Existing `Task*` surfaces are migration debt and should be renamed or removed.

## Compaction Rule

Compaction must not treat model-generated summaries as the source of truth for durable contract state.

Instead:

- rebuild the stable prefix from authoritative runtime/session state
- preserve or summarize append-only events after the rebuilt prefix
- remove superseded contract deltas that are now folded into the new prefix

This prevents drift between durable system state and the model-visible contract.

## Key Invariants

- No ambient current-time injection.
- No ambient live environment state injection.
- Durable contract data lives in the stable prefix, not in per-turn mutable blobs.
- Durable changes become append-only contract delta events until the next stable-prefix rebuild.
- Execution state does not wake the agent on intermediate status changes.
- `await` is the default execution mode.
- `detach` must be explicit.
- Detached completion is a future trigger, not an immediate follow-up turn.
- Compaction rebuilds contract state from authoritative runtime data.
- Public/runtime/proto/history/client vocabulary uses `Action` and `Execution`, not `Tool` or `Task`.

## Rejected Alternatives

- Keep a separate `volatile` prompt payload.
  - Rejected because it reintroduces mutable prompt state instead of a clean baseline plus event log.
- Inject current time in every turn.
  - Rejected because it dirties the stable prefix and weakens prompt caching.
- Inject live environment state by default.
  - Rejected because it is highly volatile and becomes stale immediately.
- Make all execution statuses model-visible.
  - Rejected because queued/running chatter creates prompt noise without improving decisions.
- Wake the agent on every execution status transition.
  - Rejected because parallel execution would cause excessive wakeups and fragmented reasoning.
