# Shared Protocol Crate And Final Cleanup

## Goal

Finish the last structural cleanup after the Action/Execution cutover by:

- introducing a shared `fathom-protocol` crate
- compiling the gRPC/proto contract exactly once
- moving shared wire-level helpers into that crate
- banning `tool` as a canonical codebase noun outside unavoidable upstream provider payload keys

The objective is to make the repository feel like one coherent post-cutover system instead of a mostly-finished migration.

## Current Problem

The runtime path already follows the new architecture, but some cleanup debt remains:

- both `fathom-server` and `fathom-client` compile the same proto independently
- both crates define their own `pb` module locally
- some internal modules still use old bridge vocabulary such as `tool_catalog` and `tool_dispatch`
- a few wire-label helpers are duplicated or live in crate-local utility modules even though they describe shared protocol enums

So the architecture is fresh, but the codebase still does not read like a cleanly separated system.

## Agreed Model

Introduce a new workspace crate:

- `fathom-protocol`

This crate is the single owner of:

- proto compilation
- generated `pb`
- gRPC client/server modules emitted from the proto
- small pure wire-level helper functions for shared enum labeling and similar contract-adjacent concerns

After cutover:

- `fathom-server` imports `fathom_protocol::pb`
- `fathom-client` imports `fathom_protocol::pb`
- `fathom-server/build.rs` no longer compiles proto
- `fathom-client/build.rs` no longer compiles proto
- crate-local `pb` modules disappear from server and client

## Boundaries

### What Belongs In `fathom-protocol`

- generated proto types
- generated gRPC service/client types
- pure protocol helpers such as:
  - `execution_status_label`
  - `execution_update_phase_label`
  - `refresh_scope_label`
  - `system_notice_level_label`

These helpers must:

- depend only on protocol enums and std
- be free of runtime/session/client UI logic
- remain deterministic and side-effect free

### What Must Not Belong In `fathom-protocol`

- prompt snapshot types
- session/runtime state
- environment registry or action metadata
- history/compaction logic
- client event-record projection
- provider adapter logic
- timestamp helpers like `now_unix_ms`
- any “shared model” that is not literally wire contract

This crate is a protocol boundary, not a shared-domain crate.

## Naming Cleanup Rule

The final noun rule is:

- canonical nouns are `Action` and `Execution`
- `tool` is banned as a codebase/domain noun outside unavoidable upstream provider payload keys

That means:

- `tool_catalog` should become `action_catalog`
- `TurnToolDispatcher` should become `TurnActionDispatcher`
- similar internal bridge names should be renamed to `action_*`

Allowed exception:

- upstream provider request/response field names that literally require `"tools"` or `"tool_choice"`

Even in those adapters, local Rust names should still prefer `action_*` where possible.

## Why This Is Better

- proto compilation happens once instead of twice
- server and client cannot drift on generated wire types
- shared enum-label behavior is centralized
- server/client crate boundaries become easier to understand
- final naming cleanup removes the last visible pre-cutover residue

## Rejected Alternatives

### Keep Per-Crate Proto Generation

Rejected because it keeps duplicate build hooks and duplicate `pb` ownership for no architectural benefit.

### Create A Broad Shared Crate

Rejected because `fathom-shared` or similar would invite runtime, prompt, and UI abstractions to leak into one generic dependency bucket.

### Allow `tool` In General Internal Names

Rejected because the codebase already standardized on `Action` / `Execution`, and lingering `tool_*` names make the migration look incomplete.

## Invariants

- `fathom-protocol` stays wire-only
- all generated gRPC/proto ownership lives there
- server and client consume the same generated contract crate
- shared helpers in that crate remain protocol-only, not runtime helpers
- `tool` naming is quarantined to unavoidable upstream provider payload keys only

## Open Issues

- none at the architectural level

Implementation details remain, but the boundary is settled.

## Next Implementation Slice

1. Add the `fathom-protocol` workspace crate and move proto compilation into it.
2. Export `pub mod pb` and the small shared wire helper set from that crate.
3. Remove local proto compilation and local `pb` modules from server and client.
4. Update imports in server and client to consume `fathom_protocol::pb`.
5. Rename remaining `tool_*` internal modules and types to `action_*`, except unavoidable upstream JSON keys inside provider adapters.
6. Run full verification and then remove any now-dead helper code left behind.
