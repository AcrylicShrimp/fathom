# Capability Domain Contract Redesign

## Design Goal

Redesign the shared `Core <-> Capability Domain` contract so capability domains are self-contained and Core no longer needs special knowledge of:

- action validation conventions
- per-domain execution entrypoints
- hard-coded domain dispatch
- harness-vs-domain argument boundaries

This redesign intentionally targets the capability integration side of the system. The top-level architecture remains:

- `Agent <-> Core <-> Capability Domain`

## Current Problem

The current shared contract is split across the wrong abstractions.

Today:

- `CapabilityDomain` owns domain metadata, recipes, initial state, and transition behavior
- `Action` owns action metadata and argument validation
- execution lives outside both traits as crate-level free functions
- Core injects harness-only controls from outside
- server registry resolves action ids, mutates schemas, validates args, strips harness fields, and dispatches execution with a hard-coded domain-id match

This means Core knows too much about capability-domain internals. The registry is not the root issue; the contract beneath it is.

## Agreed Design Direction

### 1. Remove dedicated pre-execution action validation from Core

Invalid action input should no longer be treated as a pre-execution protocol rejection in Core.

Instead:

- unknown action ids remain a Core-level rejection
- known actions with bad input become normal failed action executions
- the agent learns through ordinary execution-failure history and retries on a later turn

This removes the dedicated semantic retry path for invalid tool-call arguments.

### 2. Action listing and action execution must be unified by construction

It must not be possible for a domain to expose a listed action that cannot be selected and executed through the same contract surface.

The previous shape was vulnerable because:

- listed actions were reported separately
- validation lived on a different trait
- execution lived in free functions

The new contract must make action metadata and action execution come from the same action object.

### 3. Decode once

Action input should be decoded exactly once by the action implementation itself.

Core should not:

- re-validate raw action arguments
- deserialize arguments before dispatch
- interpret domain-specific argument semantics

### 4. Replace `schema_version()` with spec-owned schema version

The domain schema version should live in `CapabilityDomainSpec`, not as a separate trait method.

Reason:

- spec should be the single canonical contract surface
- snapshots may copy the version, but they should not source it from a separate method

### 5. Remove `initial_state` from the shared contract

`initial_state` should not remain in the base capability-domain trait.

Reason:

- the redesign moves away from Core-owned domain state
- domains should be instantiated directly as real objects rather than exposing synthetic initial JSON state through the shared trait
- build-time installation should register domain types/constructors, not state templates

Important consequence:

- the architecture must distinguish installed domain definitions from session-local domain instances

Without that distinction, a stateful domain object would be shared across sessions, which is incorrect.

### 6. `system` remains a capability domain, but with narrow injected core services

`system` cannot be fully standalone because it depends on Core.

The correct design is:

- `system` still conforms to the capability-domain contract
- `system` receives a narrow injected Core-facing interface at construction time
- other capability domains remain free from that dependency

Rejected direction:

- a giant global service bag available to all domains

Reason:

- it would quietly re-couple every domain back to Core

## Execution Semantics

### Known action id + bad input

This is a failed execution, not a Core-level rejection.

Examples:

- malformed JSON object shape
- missing required field
- wrong field type
- invalid enum/value constraint
- unsupported field combination

### Known action id + runtime problem

This is also a failed execution, but of a different kind.

Examples:

- file not found
- invalid encoding
- network timeout
- remote API failure
- non-zero process exit

### Unknown action id

This remains a Core-level rejection.

Reason:

- the agent selected something outside the installed contract
- there is no executable action object to dispatch

## Turn Execution Grouping

One agent turn may select multiple actions.

The terminology should be:

- `Action`: agent-facing selectable action definition
- `Execution`: one selected action plus its generated args
- `ExecutionSubmission`: ordered list of executions submitted as one unit to one domain instance

Only `Action` and `Execution` should be agent-visible.

`ExecutionSubmission` is strictly internal to the Core <-> Capability Domain contract.

Example selected execution order:

- `A=1`
- `B=1`
- `A=2`
- `B=2`

Core should:

1. preserve the original turn order indices
2. convert selected actions into `Execution` records
3. partition executions by capability domain
4. build one ordered `ExecutionSubmission` per target domain instance
5. submit each domain's ordered execution list to that domain
6. collect settled per-execution results from each domain
7. reorder the final result set back into original agent-selected order for history and next-turn reasoning

### Cross-Domain Ordering

Cross-domain ordering is not guaranteed within one turn.

That means:

- `A=1` and `B=1` are semantically concurrent from Core's point of view
- the agent must not rely on cross-domain within-turn sequencing

Only per-domain ordering is preserved inside one `ExecutionSubmission`.

### Per-Domain Ordering

Within one domain, the selected execution order must be preserved at the observable commit surface.

The domain may use any internal concurrency model it wants, including:

- strict serialization
- parallel work with ordered commit
- read/write-aware scheduling

But Core should observe committed results in the same per-domain order in which the agent selected them, unless the contract is explicitly expanded later.

### Domain-Owned Action-List Policy

Core must not decide how a capability domain handles its own ordered execution list.

That includes decisions such as:

- whether later actions continue after an earlier failure
- whether some actions may run concurrently internally
- whether commits are strictly serial or internally pipelined

Core only guarantees:

- the per-domain execution list is delivered in the agent-selected order
- each execution carries its original global order index in Core bookkeeping

The capability domain owns execution and commit policy for its own ordered execution list.

### Result Reordering

Domains return one result per submitted execution, in the same order as the input list.

Core maps those results back to the original action-selection order before:

- recording execution outcomes in history
- exposing the settled result set to the next agent turn

This keeps the model-visible result order aligned with the order in which the agent selected actions.

This means model-visible results are interleaved back into the original global order, not grouped by capability domain.

### Next Agent Invocation

Core should invoke the agent again only after the whole turn-local execution set has settled or moved out of foreground waiting.

This preserves the existing barrier model:

- one agent turn emits actions
- Core runs the selected action set
- the next agent turn sees the settled results

### State Ownership

This action-grouping model reinforces the state-boundary redesign:

- capability domains own their internal state and commit behavior
- Core does not compute or apply state transitions
- Core records execution results and history, not domain-managed state

### Domain Input Shape

Capability domains should receive the full ordered execution list for that domain, not single executions one-by-one from Core.

Reason:

- otherwise Core would still be implicitly controlling intra-domain execution semantics
- the domain needs the whole ordered list if it is to own continuation, concurrency, and commit policy

## Error Model

Action failures should use two top-level kinds:

- `InputError`
- `RuntimeError`

Suggested shape:

```rust
pub enum ActionError {
    InputError(ActionInputError),
    RuntimeError(ActionRuntimeError),
}

pub struct ActionInputError {
    pub code: String,
    pub message: String,
    pub details: Option<Value>,
}

pub struct ActionRuntimeError {
    pub code: String,
    pub message: String,
    pub details: Option<Value>,
}
```

Important invariant:

- stable machine-readable `code` is required
- `message` is concise human-readable guidance
- `details` is optional structured context

Without stable codes, the agent will not recover reliably from ordinary failed executions.

## Domain Factory vs Domain Instance

Removing `initial_state` means one trait is no longer enough.

The system needs two levels:

1. installed capability-domain factory
2. session-local capability-domain instance

### Domain Factory

This is registered at build time by code.

Responsibilities:

- expose domain spec
- expose action inventory
- create a session-local domain instance
- receive constructor-time dependencies

Invariant:

- `create_instance(...)` should accept a narrow session-construction context
- the context must not be the full finalized session object
- constructor-time runtime-global dependencies should still be captured by the factory itself
- session-local construction data should be limited to the minimum needed to attach the instance to one session

### Session Construction Context

The earlier zero-arg `create_instance()` assumption no longer holds once `system` becomes real.

`system` needs live session-local data, not just runtime-global services:

- current execution/runtime records
- current agent/participant identity state
- current session-local metadata

A session-start snapshot is not sufficient because these values can change during the session.

However, passing the full finalized session object into the factory would be the wrong fix. That would couple capability-domain instantiation directly to Core session internals and weaken the new boundary.

The correct widening is:

- `DomainFactory::create_instance(session_context)`
- `session_context` is narrow and Core-owned
- it should identify the session and nothing more unless a stronger need appears later

Current recommended shape:

```rust
pub struct CapabilityDomainSessionContext {
    pub session_id: String,
}
```

This keeps the domain contract session-aware without turning it into a backdoor to the entire session object graph.

### Capability Domain Instance

This is the actual stateful/self-contained object used by one engaged domain within one session.

Responsibilities:

- execute ordered execution submissions for that session/domain
- own internal state and state mutation
- own same-domain continuation/concurrency/commit policy

### Dynamic Domains

Runtime-configurable domains should not complicate the base contract.

If needed, they can be modeled as a separate installable concept such as `DynamicCapabilityDomain`, for example an MCP-backed domain.

That keeps the base contract optimized for statically installed code-defined domains.

## Proposed Shared Contract

The old `Action` trait should not survive in its current form.

The new model should be:

- domain factory owns metadata and action inventory
- session-local domain instance owns execution and internal state
- action inventory and executable selection must be tied together by construction

Suggested direction:

```rust
pub struct CapabilityDomainSpec {
    pub id: &'static str,
    pub name: &'static str,
    pub description: &'static str,
    pub schema_version: u32,
}

pub struct CapabilityActionKey(pub u16);

pub struct CapabilityActionDefinition {
    pub key: CapabilityActionKey,
    pub action_name: &'static str,
    pub description: &'static str,
    pub input_schema: Value,
}

pub trait DomainFactory: Send + Sync + 'static {
    fn spec(&self) -> CapabilityDomainSpec;
    fn recipes(&self) -> Vec<CapabilityDomainRecipe> {
        Vec::new()
    }
    fn actions(&self) -> Vec<CapabilityActionDefinition>;
    fn create_instance(
        &self,
        session_context: CapabilityDomainSessionContext,
    ) -> Box<dyn DomainInstance>;
}

pub trait DomainInstance: Send {
    async fn execute_actions(
        &mut self,
        submissions: Vec<CapabilityActionSubmission>,
    ) -> Vec<CapabilityActionResult>;
}
```

### CapabilityActionSubmission

The submission passed into action execution must already be harness-clean.

It should contain:

- action key from the installed action inventory
- normalized action arguments as raw JSON value

Suggested direction:

```rust
pub struct CapabilityActionSubmission {
    pub action_key: CapabilityActionKey,
    pub args: Value,
}
```

Intentionally absent from the domain-facing submission type:

- canonical action id
- domain id
- original turn selection index
- execution id / call id
- harness controls such as `background`

Those remain Core-owned bookkeeping and should not leak into the capability-domain contract.

Core must not pass harness-only controls like `background` through this submission.

### CapabilityActionResult

Results should be ordinary per-action outcomes, not wrapped in a separate batch result object.

The wrapper is still useful for per-action metadata that applies to both success and failure, especially execution timing.

Suggested direction:

```rust
pub struct ActionSuccess {
    pub payload: Value,
}

pub enum ActionError {
    InputError(ActionInputError),
    RuntimeError(ActionRuntimeError),
}

pub struct CapabilityActionResult {
    pub outcome: Result<ActionSuccess, ActionError>,
    pub execution_time_ms: u64,
}
```

Important invariants:

- `execute_actions(...)` must return exactly one result per submitted action
- result `i` corresponds to submission `i`
- `execution_time_ms` is domain-local wall-clock execution time for that action
- `execution_time_ms` does not include Core queueing, scheduling, or cross-domain orchestration time
- if a domain encounters a broad internal problem, it should still settle affected actions as `RuntimeError` results rather than relying on a separate batch-level error channel

### Timeout Ownership

Timeout enforcement should not be part of the capability-domain contract.

The current direction is:

- capability domains are not required to define timeout policy
- capability domains are not required to enforce timeout policy
- Core owns foreground wait policy around domain execution
- Core keeps the wait-budget table outside the shared capability-domain crate

For now, the simplest policy shape is a Core-owned wait-budget table keyed at the domain level and applied to each engaged domain instance.

This may later expand to a more granular config surface if real domains need different limits for different actions, but that should remain a Core/config concern rather than a capability-domain contract concern.

The important semantic change is:

- the budget is not a forced execution kill budget
- it is a foreground waiting budget
- if the budget expires, Core stops waiting and treats the execution as background-running
- the domain execution is allowed to continue unless it is canceled separately

### Background Hint

The old `execution_mode=detach` language should be replaced with a simpler Core-owned hint:

```json
{
  "background": true
}
```

Rules:

- `background` is optional
- default is `false`
- `background=true` means Core should start the execution directly in background rather than spending the foreground wait budget on it
- `background=true` is universally allowed and should not be rejected by action-specific policy
- `background` is a Core scheduling hint, not part of the capability-domain contract
- `background` must be stripped before action submission reaches the domain instance

This is a better fit than an execution-mode enum because the remaining decision is only whether Core waits in foreground or not.

### Execution Lifecycle

With stateful domain instances and ordered execution-submission dispatch, execution lifecycle becomes a Core-managed scheduling concern.

Current direction:

- one active `ExecutionSubmission` per domain instance
- later submissions for that same domain instance enter `queued`
- Core submits one ordered execution list to the domain instance
- Core waits up to the configured foreground wait budget
- if the domain finishes in time, the contained executions settle normally
- if the wait budget expires, Core marks the unresolved contained executions as `running_background`, triggers the agent, and lets the domain continue
- when the background execution submission later settles, Core records the final results and triggers the agent again

Suggested lifecycle states:

- `queued`
- `running_foreground`
- `running_background`
- `succeeded`
- `failed`
- `canceled`

Important invariants:

- queueing is per domain instance, not global
- Core does not cancel a domain execution merely because the foreground wait budget expired
- the next agent turn may begin after a selected execution becomes `queued` or `running_background`, not only after all executions fully settle
- if an execution is background-running, later submissions for that same domain instance remain queued behind it
- a turn is no longer blocked once none of its selected executions remain in `running_foreground`

This likely requires a protocol/history update rather than simple renaming, because the current surface only models `pending` / `running` / terminal states plus detached execution updates.

### Protocol And History Mapping

The current protocol/history surface is too tied to `await` vs `detach`.

That should be replaced with lifecycle-oriented naming.

Suggested protocol direction:

- replace generic `pending` / `running` execution statuses with:
  - `queued`
  - `running_foreground`
  - `running_background`
  - `succeeded`
  - `failed`
  - `canceled`
- stop using `detached` as the core lifecycle term
- replace the old execution-mode language with the simpler Core-owned `background` hint

Suggested execution update / prompt / history events:

- `execution_requested`
- `execution_backgrounded`
- `execution_succeeded`
- `execution_failed`
- `execution_canceled`
- `execution_rejected`

The prompt/history layer should also stop splitting terminal events into awaited-vs-detached variants.

Instead:

- whether an execution continued in background is already visible from the earlier `execution_backgrounded` event
- terminal success/failure should be modeled uniformly regardless of whether the execution finished during the initial foreground wait or later in background

For initial submission visibility:

- `execution_requested` should continue to include the current execution status
- that status may now be `queued` or `running_foreground`
- a separate `execution_started` event is not required in the first cut

This keeps the model-visible lifecycle simple:

1. execution requested
2. maybe backgrounded
3. terminal success/failure/cancel

### Action Inventory And Execution Must Be Tied By Design

The old drift problem must be prevented structurally.

Do not let `execute_actions(...)` accept raw action names and rediscover actions by string.

Instead:

- the domain factory exposes an action inventory
- Core resolves model-selected actions against that exact inventory
- Core submits `CapabilityActionKey` values derived from that inventory
- the domain instance defensively rechecks the submitted key before execution

That way:

- actions listed by the factory are the only actions executable through that domain family
- Core does not depend on raw string redispatch inside the domain
- the domain can still optimize execution across the ordered submission list

## Core Responsibilities After Redesign

Core should:

- resolve installed action by canonical id
- strip harness-only controls before action execution
- queue execution
- record success/failure outcome
- reorder settled cross-domain results back into original global order

Core should not:

- validate domain-specific arguments
- decode domain-specific arguments
- branch on capability-domain id to discover execution behavior
- apply domain state transitions

## Implications For Registry Redesign

Once this contract exists, the registry becomes simpler:

- catalog installed domain factories and actions
- resolve action ids to domain factories and action definitions
- expose summaries and tool definitions

The registry should no longer:

- own validation of domain-specific input
- dispatch execution with a hard-coded domain match
- own domain state templates or transitions

## Implications For Agent Retry Behavior

The current model-adapter semantic retry path for invalid action arguments becomes unnecessary.

Instead:

- invalid input becomes an ordinary failed execution
- the failure enters session history
- the agent sees the structured failure and retries in a later turn if appropriate

This matches modern agent-tool loops more closely.

## System Domain Scope

The redesigned `system` capability domain should stay narrow.

It should expose Core-owned runtime and execution inspection, not duplicate stable-prefix prompt material and not expose other capability domains' internal state.

It should inspect live session-local state by querying the session actor through explicit commands, not by directly reading `SessionState` or other Core internals.

Current direction:

- drop `get_time` from `system`
- do not keep broad session/profile/capability-description actions if they only restate stable-prefix baseline material
- keep `system` focused on execution/runtime inspection needed after the new queued/foreground/background lifecycle redesign

Minimal action set:

- `list_executions`
- `get_execution`
- `read_execution_input`
- `read_execution_result`

Likely later:

- `cancel_execution`

### Locked System Inspection Model

#### Visible Units

`system` should expose single `Execution` records, not internal `ExecutionSubmission`s.

- `Execution`: one action selected by the agent plus its args
- `ExecutionSubmission`: internal ordered list of executions submitted to one domain instance

`ExecutionSubmission` remains strictly internal to Core and the capability-domain contract.

#### Execution Lifecycle Visibility

Visible lifecycle is per `Execution`, but while unresolved it is derived from the owning `ExecutionSubmission`.

- queued submission => all contained unresolved executions are `queued`
- foreground-running submission => all contained unresolved executions are `running_foreground`
- background-running submission => all contained unresolved executions are `running_background`
- when the submission finishes, each execution receives its own terminal result

V1 does not require partial per-execution progress reporting inside a running submission.

#### `system__list_executions`

Purpose:

- discover execution ids
- inspect compact execution summaries

Arguments:

- cursor-based pagination
- opaque `cursor`
- optional `limit`
- optional single-value filters:
  - `state`
  - `action_id`

Default behavior:

- includes all executions when no filter is provided
- ordered by descending global execution order
- default `limit = 20`
- max `limit = 100`
- values above the max are clamped
- `state` filter is exact equality on one state value
- `action_id` filter is exact equality on one action id

Response item shape:

- `execution_id`
- `action_id`
- `state`

Response envelope:

- `executions`
- `next_cursor`
- `prev_cursor`

#### `system__get_execution`

Purpose:

- inspect one execution in detail

Response shape:

- direct execution object
- not wrapped in an outer `execution` field

Returned fields:

- `execution_id`
- `action_id`
- `state`
- `input` payload descriptor
- optional `result` payload descriptor
- `execution_time_ms` for terminal states only

No timestamps should be exposed.

No separate `background_requested` field should be exposed because state already covers observable background status.

This should be one stable execution object shape:

- some fields are always present
- some fields are state-dependent

Specifically:

- `input` is always present
- `result` is omitted until a result payload actually exists
- when present, `result` is the payload descriptor itself
- `execution_time_ms` is present only for terminal states such as `succeeded`, `failed`, and future `canceled`

#### Payload Descriptor Shape

Input payload is always available.

Result payload may be absent until materialized.

When available, payload preview is based on one shared Core-owned preview threshold for both input and result.

There are two normalized descriptor forms:

1. complete view
   - `total_size`
   - `content`

2. truncated view
   - `total_size`
   - `prefix`
   - `prefix_size`
   - `suffix`
   - `suffix_size`

Payloads are serialized to a minified string form first. JSON payloads should be minified and stringified consistently before previewing or ranged reading.

#### Unified Payload Slice Response

`system__read_execution_input` and `system__read_execution_result` should use the same success shape:

- `total_size`
- effective `offset`
- effective `limit`
- `content`

This is the normalized ranged-read response shape for payload reading.

#### `system__read_execution_input`

Purpose:

- read the serialized execution input payload by byte range

Args:

- `execution_id`
- optional `offset`
- `limit`

Response:

- unified payload slice response

Semantics:

- input payload is always readable
- reads are byte-range based only
- ranges are clamped rather than rejected
- `offset` defaults to `0`
- `limit` is required
- `limit` is clamped to a Core-owned maximum byte count

#### `system__read_execution_result`

Purpose:

- read the serialized execution result payload by byte range

Args:

- `execution_id`
- optional `offset`
- `limit`

Response when available:

- unified payload slice response

Response when not yet materialized:

- `InputError`

Semantics:

- reads are byte-range based only
- ranges are clamped rather than rejected
- `offset` defaults to `0`
- `limit` is required
- `limit` is clamped to a Core-owned maximum byte count
- failure details stay inside the result payload itself rather than duplicated as top-level execution summary fields

### Time Moves To A Separate Domain

Current time and time arithmetic should move out of `system`.

Reason:

- current time is not really privileged system inspection
- date/time arithmetic is a separate capability shape from execution/runtime inspection
- future operations such as "what exact date is next week" need a more intentional time capability surface with clear timezone/reference-time semantics

That future time domain can cover:

- current time lookup
- exact relative-date resolution
- explicit date/time arithmetic and normalization
- timezone-aware temporal calculations

### System Inspection Service Boundary

The `system` domain should not depend on `Runtime` directly.

Instead:

- `SystemDomainFactory` should receive a dedicated narrow trait such as `SystemInspectionService`
- the trait should be injected at factory construction time
- the trait should expose only the queries needed by:
  - `list_executions`
  - `get_execution`
  - `read_execution_input`
  - `read_execution_result`

The service should return already shaped Core inspection records rather than forcing `system` to reconstruct execution views from many smaller queries.

This keeps `system` privileged but still isolated from broad Core internals.

## Open Issues

- Whether `FinalizedAction` should continue to store `args_json` verbatim or store normalized/cleaned arguments only.
- Exact protocol/history representation for `queued`, `running_foreground`, and `running_background`.
- Exact method set and record types inside `SystemInspectionService`.

## Hard-Cutover Implementation Plan

This redesign should be landed as a hard cutover, not as a compatibility bridge between the old and new capability contracts.

Recommended implementation order:

### Phase 1. Replace The Shared Capability Contract

Target:

- `fathom-capability-domain`

Work:

- remove the old `Action` trait
- remove `initial_state`
- remove `apply_transition`
- remove Core-facing snapshot/transition types from the base contract
- introduce:
  - `DomainFactory`
  - `DomainInstance`
  - `CapabilityDomainSessionContext`
  - `CapabilityActionKey`
  - `CapabilityActionDefinition`
  - `CapabilityActionSubmission`
  - `CapabilityActionResult`
  - `ActionError` / `InputError` / `RuntimeError`
- keep the contract free of harness controls such as `background`

Result:

- the shared crate defines the new Core <-> Capability Domain boundary
- the workspace is expected to break until downstream code is ported

### Phase 2. Redesign Core Execution Records And Lifecycle

Target:

- Core execution model, protocol, history, prompt event types

Work:

- keep `Execution` as the single-action user/agent-facing runtime record
- introduce internal `ExecutionSubmission` as the ordered domain-submission unit
- introduce the new lifecycle:
  - `queued`
  - `running_foreground`
  - `running_background`
  - `succeeded`
  - `failed`
  - `canceled`
- replace `execution_mode=detach` with the Core-owned optional `background=true|false` hint
- make foreground wait budget a Core scheduling concern rather than domain timeout policy
- make turn barrier semantics depend on unresolved foreground submissions, not “all executions fully settled”
  - a queued foreground submission still blocks the turn if it is waiting behind earlier same-domain work

Result:

- Core and protocol stop speaking in awaited-vs-detached terms
- lifecycle and scheduling become explicit before domains are reattached

### Phase 3. Rebuild The Registry As A Catalog Only

Target:

- capability registry / bootstrap / resolution

Work:

- make the registry a catalog of installed `DomainFactory`s and action definitions
- remove execution dispatch from the registry
- remove domain-input validation from the registry
- remove timeout/mode policy from the registry
- keep only:
  - installation catalog
  - action-id resolution
  - prompt/tool-definition projection

Result:

- registry becomes metadata/catalog infrastructure only

### Phase 4. Replace Domain Actors With Instance-Owned Execution

Target:

- session runtime execution path

Work:

- instantiate one engaged `DomainInstance` per engaged domain per session
- create each instance through `DomainFactory::create_instance(session_context)`
- queue execution requests per domain instance
- allow one active execution per domain instance
- submit ordered `Vec<CapabilityActionSubmission>` directly to the instance
- record per-action ordered results and restore original cross-domain order for the agent
- keep Core bookkeeping outside the domain-facing contract

Result:

- Core no longer computes domain transitions
- Core no longer redispatches domain execution by hard-coded domain id

### Phase 5. Port Capability Domains To The New Contract

Target:

- filesystem
- shell
- brave_search
- jina
- system

Work:

- rewrite each domain as a `DomainFactory` plus session-local `DomainInstance`
- move arg decoding into execution and return `InputError` / `RuntimeError` instead of pre-execution validation failures
- remove old free `execute_action(...)` entrypoints and old per-action trait wiring

Result:

- every capability domain becomes self-contained under the new boundary

### Phase 6. Rebuild The System Domain On Top Of New Core Inspection APIs

Target:

- `system`

Work:

- drop legacy broad baseline-duplication actions
- implement the narrow execution/runtime inspection surface:
  - `list_executions`
  - `get_execution`
  - `read_execution_input`
  - `read_execution_result`
- optionally add `cancel_execution` later
- make the system domain depend only on:
  - factory-time injected `SystemInspectionService`
  - session-construction context identifying the target session

Result:

- `system` becomes a thin privileged inspection layer, not a dumping ground for unrelated runtime reads

### Phase 7. Clean Up Prompt, History, And Legacy Surfaces

Target:

- agent-facing prompt contract
- history schema
- diagnostics
- payload lookup helpers

Work:

- remove awaited-vs-detached event variants
- replace them with lifecycle-oriented execution events
- remove legacy `execution_mode` / `detach` language from prompt/harness text
- align payload lookup with execution-only addressing
- delete replaced compatibility paths and dead code

Result:

- the agent-facing surface matches the new execution model end-to-end

## First Implementation Slice

The best first coding slice is:
