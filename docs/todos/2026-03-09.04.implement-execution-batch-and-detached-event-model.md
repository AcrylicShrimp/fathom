## Context
This slice derives from the finalized execution design in docs/designs/2026-03-09.00.agent-prompt-context-and-execution-model.md. The runtime currently exposes older tool lifecycle and task-flow behavior that does not match the agreed batch-settle semantics, `await` / `detach` mode handling, or the finalized six execution event types.

The implementation needs to align capability metadata, tool dispatch behavior, agent wakeup policy, and model-visible events with the agreed execution model.

## Why It Is Needed
Execution semantics are central to the new harness contract. The agent must be able to choose legal tool modes, rely on one wakeup after batch settle, and reason over a small, stable execution-event surface. Without this work, the new prompt model and the runtime behavior would contradict each other.

## Completion Criteria
- [ ] Represent tool `mode_support` using the agreed `await_only` / `await_or_detach` model.
- [ ] Make `await` the default execution mode and require explicit `detach`.
- [ ] Implement batch-settle behavior so the agent resumes once after all awaited calls settle and detached calls are accepted or rejected.
- [ ] Emit the agreed model-visible execution event types: awaited success/failure, detached success/failure, execution_detached, and execution_rejected.
- [ ] Suppress intermediate queued/running status chatter from model-visible wakeups.
- [ ] Coalesce later detached completions into future trigger batches.
- [ ] Add focused tests for batch wakeup semantics, illegal detach rejection, and detached completion triggers.

## Detailed Plan
1. Extend capability/tool metadata with the new mode-support contract.

3. Replace current tool lifecycle event emission with the six settled execution event types.
