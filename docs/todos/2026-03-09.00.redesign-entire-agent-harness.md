## Context
The current agent harness in `fathom-server` is functional but tightly coupled across turn coordination, prompt assembly, provider streaming, tool dispatch, event emission, and diagnostics. A full analysis pass and redesign draft were completed on 2026-03-09 before implementation began.

Key issues identified:
- `session/engine/turn.rs` mixed orchestration, provider callbacks, task queueing, event emission, and diagnostics capture.
- The OpenAI request exposed the global tool registry instead of a session-scoped engaged-tool catalog.
- Prompt generation was rebuilt in more than one place and history handling relied on dynamic event payloads.
- Internal system notices and turn failures surfaced as assistant outputs, forcing client-side filtering.
- Compaction was modeled in session state but not active.

This TODO tracks the remaining harness redesign work. The target is a simpler, correct harness, not a separate abstraction-only phase for hypothetical prompt or delta redesigns.

## Why It Is Needed
A dedicated redesign is needed to improve correctness, maintainability, observability, and long-session behavior before further feature work expands the harness surface area. Evidence from the analysis includes:
- engaged-tool enforcement originally happened after model generation instead of constraining provider-visible tools up front;
- turn execution relied on callback glue and shared mutable capture inside one path;
- history handling depended on dynamic event names and payloads;
- internal notices were mixed with user-visible assistant output;
- session compaction was not active.

Without this refactor, provider integration work, prompt changes, richer tool lifecycle events, and long-session context management become harder to implement safely.

## Completion Criteria
- [x] Finish the dedicated turn harness architecture so turn coordination, prompt assembly, model adaptation, tool dispatch, and invocation journaling are clearly separated.
- [x] Enforce session-scoped engaged-tool visibility in provider requests instead of relying on post-hoc task rejection.
- [x] Replace stringly-typed history handling in the harness path with typed history events or an equivalent structured representation.
- [x] Separate user-visible assistant output from internal diagnostics/tool lifecycle notices at the server event model level.
- [x] Define and land a provider-neutral model adapter boundary with OpenAI as one implementation.
- [x] Activate a concrete compaction strategy or summary mechanism suitable for long-running sessions.
- [ ] Add tests that cover event ordering, engaged-tool enforcement, retries, failure handling, prompt assembly behavior, and delta-stream behavior.
- [x] Update architecture documentation to match the new harness design.

## Detailed Plan
1. Finish the harness structure.
- Preserve the single session actor, barrier scheduling, task model, and per-environment deterministic commit ordering.
- Keep the decomposed turn path centered on `TurnCoordinator`, prompt assembly, model adaptation, session-scoped tool catalog construction, tool dispatch, and invocation journaling.
- Remove remaining coupling where the orchestrator still depends on provider-specific behavior.

2. Finish execution-path cleanup.
- Ensure prompt assembly is built once per attempt and reused for diagnostics.
- Keep the tool catalog derived from engaged environments only.
- Keep provider output flowing through typed harness events instead of ad hoc callback glue.

3. Finish memory and event cleanup.
- Keep internal/system notices and tool lifecycle distinct from assistant-facing output.
- Use typed history records end to end.
- Keep session compaction/summarization active and covered by tests.

4. Verify and document.
- Add targeted tests for scheduling, retries, tool visibility, prompt assembly, delta sequencing, event sequencing, and prompt budgeting.
- Update `ARCHITECTURE.md` and related module docs to reflect the final harness design.
